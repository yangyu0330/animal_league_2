# 학과 과제 압박 랭킹 서비스 MVP 계획서

## Summary
- 목표는 `학생이 과제 때문에 힘들 때 학과를 탭해 압박도를 올리는` 전국 대학 모바일 웹 MVP를 6주 안에 출시하는 것이다.
- 공개 단위는 `학교명 + 학과명`이며, 교수 개인 정보는 다루지 않는다.
- 메인 랭킹 기준은 `총 클릭 수`만 사용한다. 공정성 보정 지표는 MVP에서 제외한다.
- 참여 방식은 `소셜 로그인 + 학교 선택`이며, 댓글/게시판 없이 `클릭만` 제공한다.
- 초기 데이터는 `상위 대학 시드 데이터 + 사용자 추가 등록`으로 시작한다.
- 학과 캐릭터는 AI 생성이 아니라 `계열별 기본 마스코트 템플릿`을 사용하고, 클릭 수에 따라 부담 상태만 변화시킨다.

## Key Changes
### 제품 구조
- 첫 진입: 서비스 설명, 랭킹 미리보기, 1탭 참여 방식 안내
- 홈: 전국 TOP 10, 오늘 급상승 학과, 학교 필터, 검색 진입
- 검색/등록: 학교 선택 후 학과 검색, 없으면 새 학과 추가
- 학과 상세: 학과명, 학교명, 총 클릭 수, 학생 스택 수, 압박 단계, 메인 클릭 버튼, 공유 버튼
- 내 활동: 오늘 내가 누른 학과, 최근 클릭 기록
- 관리자 기능은 별도 페이지를 만들지 않고 MVP에서는 Supabase 대시보드로 운영한다

### 게임 규칙
- 메인 지표는 총 클릭 수이며 랭킹은 실시간으로 정렬한다
- 학생 스택 수는 `floor(total_clicks / 1000)`로 계산한다
- 압박 단계는 총 클릭 수 구간으로 5단계 고정한다
  - 0: 0~999
  - 1: 1,000~4,999
  - 2: 5,000~9,999
  - 3: 10,000~24,999
  - 4: 25,000+
- 클릭은 완전 자유 클릭으로 두되, 서버는 오토클릭만 감지해 제외한다
- 오토클릭 감지는 계정, 디바이스 해시, IP 기준 burst 패턴으로 처리하고 의심 클릭은 저장하되 랭킹 합산에서 제외한다

### MVP 구현 방식
- 프론트엔드는 `Next.js App Router` 기반 모바일 웹으로 구성한다
- 백엔드는 `Supabase`를 사용한다
  - Auth: Google OAuth 1종만 MVP 범위에 포함
  - DB: Postgres
  - Storage: 마스코트 템플릿 및 공유 썸네일 자산 저장
- 배포는 `Vercel`, 모니터링은 `Sentry`, 제품 분석은 `PostHog`로 고정한다
- 시드 데이터는 `대학 20개 내외 + 각 대학 주요 학과 8~12개`를 CSV로 넣고, 사용자 등록으로 확장한다
- 학과 등록은 즉시 반영하되 `학교+정규화된 학과명` 중복 검사로 막고, 유사 중복은 운영자가 수동 병합한다
- 마스코트는 계열별 8종 템플릿으로 시작한다
  - 공학, 자연과학, 인문, 사회과학, 경영/경제, 예술/체육, 교육, 보건/의학
- 각 마스코트는 압박 단계별 표정/배경만 다르게 노출한다

### 데이터 모델 / 공개 인터페이스
- `User`
  - `id`, `provider`, `nickname`, `selected_school_id`, `created_at`
- `School`
  - `id`, `name`, `region`, `is_seeded`
- `Department`
  - `id`, `school_id`, `name`, `normalized_name`, `category`, `template_id`, `total_clicks`, `accepted_clicks`, `pressure_level`, `created_by`, `created_at`
- `ClickEvent`
  - `id`, `user_id`, `department_id`, `device_hash`, `ip_hash`, `accepted`, `reason`, `created_at`
- `DepartmentDailyStat`
  - `date`, `department_id`, `accepted_clicks`
- API/서버 핸들러
  - `GET /api/rankings?scope=national|school&schoolId=`
  - `GET /api/departments/search?q=&schoolId=`
  - `POST /api/departments`
  - `GET /api/departments/:id`
  - `POST /api/departments/:id/click`
  - `GET /api/me/activity`
- 응답 규칙
  - 랭킹 응답은 `rank`, `departmentName`, `schoolName`, `totalClicks`, `stackCount`, `pressureLevel`
  - 클릭 응답은 `accepted`, `newTotalClicks`, `stackCount`, `pressureLevel`

## 3인 팀 역할 / 6주 일정
### 역할 분담
- 팀원 A: 프론트엔드 오너
  - App Router 구조, 홈/검색/상세/내 활동 화면, 모바일 UI, 공유 플로우
- 팀원 B: 백엔드 오너
  - Supabase 스키마, OAuth, 클릭 수집 API, 오토클릭 감지, 랭킹 쿼리
- 팀원 C: 풀스택/프로덕트 오너
  - 시드 데이터 제작, 마스코트 템플릿, 검색/등록 UX, 분석도구, QA 총괄

### 주차별 계획
- 1주차
  - 화면 구조 확정, DB 스키마 확정, 디자인 토큰 정의, 시드 데이터 범위 확정
- 2주차
  - 로그인, 학교 선택, 홈 랭킹 읽기, Supabase 스키마/시드 적재 완료
- 3주차
  - 학과 검색, 학과 등록, 학과 상세, 기본 클릭 집계 연결
- 4주차
  - 자유 클릭 처리, 오토클릭 감지, 오늘 급상승/내 활동, 공유 링크 구현
- 5주차
  - 마스코트 단계 변화, 예외처리, 성능 점검, 이벤트 분석/에러 모니터링 연결
- 6주차
  - 클로즈드 테스트, 버그 수정, 랭킹 정확도 검증, 배포 및 운영 체크리스트 확정

## Test Plan
- 로그인 후 학교 선택을 완료해야 홈 랭킹과 검색이 정상 노출되어야 한다
- 시드 학과 검색이 학교 필터와 함께 정확히 동작해야 한다
- 없는 학과 등록 시 중복 검사 후 즉시 상세 페이지로 이동해야 한다
- 클릭 1회 시 학과 총 클릭 수와 랭킹이 즉시 갱신되어야 한다
- 클릭 수가 1,000을 넘는 시점에 학생 스택 수가 정확히 1 증가해야 한다
- 압박 단계가 구간 기준대로 정확히 바뀌고 UI 상태도 같은 기준을 따라야 한다
- 짧은 시간 반복 탭과 일반 사용자의 빠른 탭을 구분해 오토클릭만 제외해야 한다
- 학교별 랭킹과 전국 랭킹이 같은 데이터 원본을 기준으로 일관되게 계산되어야 한다
- 공유 링크로 진입한 사용자가 해당 학과 상세로 바로 진입해야 한다
- 모바일 화면에서 한 손 조작 기준으로 홈, 검색, 클릭, 공유 흐름이 깨지지 않아야 한다

## Assumptions
- MVP 범위에는 교수, 강의명, 과목평, 댓글, 자유게시판을 넣지 않는다
- 소셜 로그인은 MVP에서 `Google 1종`만 지원하고, Kakao/Apple은 후속 범위로 둔다
- 즐겨찾기 기능은 제외하고 `내 최근 클릭 기록`만 제공한다
- 운영도구는 별도 CMS 없이 Supabase 콘솔 기반으로 시작한다
- 총 클릭만 메인 지표로 사용하므로 학과 규모 편향은 존재하며, `클릭 밀도`는 후속 지표로 남긴다
- 사용자 등록 학과는 즉시 공개하되, 중복 병합과 비정상 데이터 정리는 수동 운영으로 처리한다
